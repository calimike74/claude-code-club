# 04 — Shipping and Beyond

## This Document Is Not About Tools

Everything else in this collection is about what to build and what to build with. This document is about the things that make all of that actually matter.

---

## Ship Publicly

The single biggest gap between "I'm learning to code" and "I built something" is a URL you can send to someone. Every student in the club should leave with at least one live project with a real domain name.

Not on localhost. Not in a GitHub repo nobody will ever look at. A link they can text to their mum, put on their CV, or drop in a UCAS personal statement.

The act of deploying changes how they think about what they're building. It stops being homework and starts being real.

**Practical steps:**
- Deploy to Vercel from day one, even if the project is half-finished
- A live URL with a basic landing page is better than a perfect project that only exists locally
- Once they have a custom domain (free .me via Namecheap + GitHub Student Pack), point it at their Vercel project immediately

---

## Document As You Build

Not for the teacher. Not for a mark scheme. For themselves in six months when they've forgotten how they did it.

Every project repo should have a README that covers:

1. **What it does** — one or two sentences
2. **Why they built it** — the problem it solves
3. **How to use it** — link to the live deployment
4. **What they'd do differently** — honest reflection
5. **Tech used** — Claude Code, React, Supabase, etc.

This takes ten minutes and is the difference between a portfolio piece and an abandoned folder. Claude Code can help them write it, but they need to know it matters.

## The University-Bound Students

### Their advantage is temporary
Right now they're ahead because they can use Claude Code and most people can't. By Christmas of their first year, coursemates will have caught up on the tools. What won't be easily replicated is the **instinct to build** — to see a problem and think "I could make something for that" rather than "someone should make an app for that."

That's the real thing the club teaches, and it's worth saying out loud. Tool literacy is the vehicle. The builder mindset is the destination.

### They should tell the story
When those students arrive at university and casually mention they've been using agentic AI to build and deploy web apps since lower sixth, that's a story. It differentiates them immediately — in seminars, in group projects, in conversations with lecturers.

Make sure they know how to tell it. Not "I used ChatGPT" — that's generic. "I built and deployed a web application using Claude Code with a Supabase backend and Gemini API integration" — that's specific and impressive.

### The toolkit transfers completely
Everything they've learned — Claude Code, Supabase, Vercel, the whole stack — works identically at university. They arrive knowing how to build and deploy web apps, which puts them ahead of most freshers, even those on computer science courses.

## The Work-Bound Students

### A GitHub profile that tells a story
Not fifty empty repos. Three to five projects with:
- Clean READMEs explaining what each project does
- Live deployment links
- Commit histories that show iterative development (not one massive dump)

An employer looking at a GitHub profile can tell in thirty seconds whether someone builds things or just talks about building things.

### A deployed project beats a CV line
For any job where this matters — tech, finance, consulting, anything digital — being able to say "here's something I built, and here's the URL" is more useful than a reference letter. It's proof of capability that can't be faked.

## Peer Teaching

The strongest club members should be paired with the less confident ones before they leave. Not as formal mentoring — just "show James how you set up Supabase auth."

Why this matters:
- The student doing the teaching learns more than the one being taught
- It creates a culture where helping is normal rather than something only the teacher does
- If any of them end up at the same university, that relationship carries forward
- It's a skill (explaining technical concepts clearly) that's valuable in every career

---

## Open Source Contribution

Nobody has told most students this exists as a thing they can do. They don't need to contribute to Linux. But:

- Finding a small project on GitHub
- Fixing a typo in documentation
- Adding a feature to something with fifty stars

That's a commit on their profile from a project they didn't create, which signals to anyone looking that they understand collaboration.

Claude Code makes this much more accessible. They can clone a repo, understand the codebase with Claude's help, and make a meaningful contribution without being an expert.

**One merged pull request on someone else's project is worth ten personal repos.**

### How to find a first contribution:
- Search GitHub for `good first issue` labels
- Look at projects they actually use
- Documentation fixes are always welcome and low-risk
- Translation contributions if they speak another language

## Demo Day

Before they leave, run a demo day. Even if it's informal — twenty minutes in a classroom, each student shows what they built, takes questions.

### Why it works:
- The pressure of knowing they'll present focuses the work enormously
- It gives them practice articulating what they made and why
- That's exactly what they'll need in interviews, personal statements, and university presentations
- It creates a deadline, and deadlines ship products

### How to run it:
- 3–5 minutes per student maximum
- Show the live project (not slides about the project)
- Explain the problem it solves
- Take one or two questions
- If possible, get a member of SLT or a governor to attend — it signals the school values what they've done
- Record short clips (with permission) for LinkedIn or the school's marketing

## Write Them a Reference Paragraph

Not a full reference — just a paragraph kept on file that can be dropped into any reference you're asked to write. Something specific:

> "Built and deployed a web application using modern development tools including React, Supabase, and AI-assisted development. Demonstrated initiative in identifying a real problem and building a working solution independently. Comfortable with version control, deployment pipelines, and iterative development — skills typically associated with professional software teams."

Most referees write generic things. A specific, technical paragraph stands out. You're probably the only teacher in their school who can write this credibly.

Adapt per student based on what they actually built. Keep a one-line note for each student: "James — built the lost property system with Gemini image recognition." That's enough to write a strong paragraph months later when a reference is requested.

---

## The One Thing That Matters Most

Your instinct to support, encourage and push is right. But the most important thing you can do in the last three months isn't teaching them a new tool or technique. It's making sure they **finish something**.

The graveyard of student projects is enormous, and almost all of them died not because the student lacked ability but because nobody said "stop adding features and ship it."

Be the person who says that.

The student with one finished, deployed, imperfect project has learned more than the student with three ambitious half-built ones.

---

## What You're Building Here

A Claude Code club for sixth formers, with students deploying real applications, is not happening in most schools. When those students talk about what they did at school, this will be one of the things that sticks.

The tools will change. Claude Code might not exist in five years. But the habit of seeing a problem and building a solution — that's permanent. That's what you're actually teaching.
